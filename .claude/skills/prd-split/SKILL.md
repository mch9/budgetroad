# PRD Split Skill

하나의 큰 프로덕트 아이디어를 **경험 단위**로 쪼개고,
각 경험을 `/prd-collab`으로 하나씩 완성해나가는 워크플로우.

## When to Use

- 사용자가 `/prd-split`을 입력했을 때
- 사용자가 "이런 서비스를 만들고 싶어요"라며 큰 아이디어를 던졌을 때

## 핵심 개념: 경험 단위란?

```
❌ 화면 단위로 쪼개면                    ✅ 경험 단위로 쪼개면
┌────────────────────┐              ┌────────────────────────────────┐
│ PRD 1: 홈 화면      │              │ PRD 1: "나를 진단받는 경험"      │
│ PRD 2: 결과 화면    │              │  질문 시작 → 응답 → 결과 확인    │
│ PRD 3: 리포트 화면  │              │  (시작과 끝이 있는 하나의 흐름)   │
│ PRD 4: 결제 화면    │              │                                │
│ (서로 연결이 안 됨)  │              │ PRD 2: "유료 리포트 구매 경험"    │
└────────────────────┘              │  결과 확인 → 리포트 미리보기 →   │
                                    │  결제 → 리포트 열람              │
                                    │                                │
                                    │ PRD 3: "전문가 상담 예약 경험"    │
                                    │  결과 기반 추천 → 상담사 선택 →   │
                                    │  일정 선택 → 예약 완료           │
                                    └────────────────────────────────┘
```

**경험 단위** = 유저가 시작점에서 출발하여 끝점에 도달하는 하나의 완결된 플로우.
- 반드시 **시작**과 **끝**이 있어야 한다
- 유저가 "하나의 일을 해냈다"고 느낄 수 있어야 한다
- 화면이 여러 개 걸쳐도 하나의 경험이면 하나의 PRD
- 하나의 화면 안에 독립된 경험이 2개면 PRD도 2개

## Instructions

### Step 1: 이 단계가 뭔지 설명하기

아래 내용을 출력하세요:

---

## 아이디어를 경험 단위로 쪼개기

```
당신의 큰 아이디어 🚀
"간호사 커리어 전환 진단 웹앱"
        │
        ▼
  ┌─────────────┐
  │  경험 1       │ ← "나를 진단받는 경험"
  │  경험 2       │ ← "유료 리포트 구매 경험"
  │  경험 3       │ ← "전문가 상담 예약 경험"
  │  ...         │
  └─────────────┘
        │
        ▼  각 경험마다 /prd-collab 실행
  ┌─────────────┐
  │  PRD 1 완성 ✅ │
  │  PRD 2 완성 ✅ │
  │  PRD 3 완성 ✅ │
  └─────────────┘
```

**쉽게 말하면:**
"집을 짓고 싶어요"라고 하면 한 번에 다 설계하기 어렵잖아요.
그래서 "현관에서 거실까지의 동선", "주방에서 식사까지의 동선"처럼
**사람이 실제로 움직이는 경로**별로 나눠서 하나씩 설계하는 거예요.

- header: "우주기획 던지기 🚀"

### Step 4: 경험 단위로 쪼개기

사용자의 우주기획을 분석하여 **경험 단위**로 쪼갭니다.

쪼개기 기준:
1. **시작과 끝이 명확한가?** — 유저가 "시작"해서 "완료"에 도달하는 흐름
2. **독립적으로 가치가 있는가?** — 이 경험만 있어도 유저가 뭔가를 얻는가
3. **다른 경험 없이도 테스트 가능한가?** — MVP로 이것만 먼저 만들어도 되는가

결과를 아래 형식으로 정리:

```
## 경험 분해 결과

### 🟢 MVP (첫 번째로 만들 것)
1. **{경험명}** — {한 줄 설명}
   - 시작: {진입점}
   - 끝: {완료 상태}
   - 핵심 가치: {유저가 얻는 것}

### 🟡 Phase 2 (MVP 이후)
2. **{경험명}** — {한 줄 설명}
   - 시작: {진입점}
   - 끝: {완료 상태}
   - 핵심 가치: {유저가 얻는 것}
   - 의존: {선행 경험}

### 🔴 Phase 3 (나중에)
3. **{경험명}** — {한 줄 설명}
   ...
```

### Step 6: 경험 로드맵 파일 생성

합의된 경험 분해를 `docs/prd/ROADMAP.md`로 저장합니다:

```markdown
# Experience Roadmap

## 프로덕트 비전
{사용자의 우주기획 요약}

## 경험 단위 분해

### 🟢 MVP
| # | 경험명 | 설명 | PRD 상태 |
|---|--------|------|----------|
| 1 | {이름} | {설명} | ⬜ 미작성 |

### 🟡 Phase 2
| # | 경험명 | 설명 | 의존 | PRD 상태 |
|---|--------|------|------|----------|
| 2 | {이름} | {설명} | #{의존번호} | ⬜ 미작성 |

### 🔴 Phase 3
| # | 경험명 | 설명 | 의존 | PRD 상태 |
|---|--------|------|------|----------|
| 3 | {이름} | {설명} | #{의존번호} | ⬜ 미작성 |

---
**Last Updated**: YYYY-MM-DD
```

### Step 7: 첫 번째 경험 PRD 작성 시작

ROADMAP.md 저장 후 안내:

```
✅ 경험 로드맵이 완성되었습니다!

📋 총 {N}개의 경험으로 분해
   🟢 MVP: {M}개
   🟡 Phase 2: {K}개
   🔴 Phase 3: {L}개

이제 첫 번째 경험부터 상세 기획을 시작합니다.
```

그리고 바로 `/prd-collab`을 호출하여 MVP 첫 번째 경험의 PRD 작성을 시작합니다.

### Step 8: 각 경험 완료 시 로드맵 업데이트

하나의 `/prd-collab`이 완료될 때마다:
1. `docs/prd/ROADMAP.md`의 해당 경험 상태를 `✅ 완료`로 업데이트
2. 다음 경험을 안내하고, 사용자 확인 후 다음 `/prd-collab` 호출

```
✅ "{경험명}" PRD 완성!

남은 경험:
  ⬜ {다음 경험명} — {설명}
  ⬜ ...

다음 경험 기획을 시작할까요?
```

## Rules

- 경험은 **화면 단위가 아니라 유저 플로우 단위**로 쪼갠다
- MVP는 최소한으로 — 가장 핵심적인 경험 1~2개만
- 각 경험의 PRD는 반드시 `/prd-collab` 워크플로우로 작성
- 한 번에 하나의 경험만 기획 (병렬 진행 금지)
- ROADMAP.md는 항상 최신 상태 유지
