0. 한 줄 요약
현재 예식장 소계는 "보증인원 초과 시 대관료 면제"라는 호텔 전용 모델을 모든 베뉴에 적용하고 있다. 일반 웨딩홀·하우스·스몰 베뉴는 대관료를 별도 청구하므로, 이 케이스에서 core 예산이 150~230만원 과소 계상된다. 해결은 STAGE 4가 이미 산출하는 추천 식장 형태를 베뉴 타입으로 자동 매핑하고, 베뉴 타입별로 식대·대관료 공식을 분기하는 것이다. 사용자 추가 질문은 필요 없다.

A. 베뉴 타입 판단 방법
베뉴 타입을 알아야 대관료 분기가 가능하다. 입력 데이터에 베뉴 타입 질문이 없으므로 판단 방법을 정해야 한다.
옵션 비교
   옵션 방법 장점 단점 구현 비용     (a) Q15 신규 질문 사용자에게 호텔/일반/하우스/스몰/모름 직접 질문 가장 정확 14문항도 이미 김 → 이탈 위험. 질문 추가는 보수적이어야 함 중 (질문·UI·가중치)   (b) STAGE 4 자동 매핑 이미 산출 중인 추천 식장 형태를 베뉴 타입으로 변환 추가 질문 0, 데이터 이미 있음, 즉시 적용 추천이 "현실 선택"과 다를 수 있음(원하는 형태 ≠ 계약 형태) 낮음   (c) M1+M5+유형 휴리스틱 예산·지역·유형으로 추론 질문 없음 STAGE 4와 로직 중복, 규칙 충돌 관리 부담 중   (d) 혼합 1순위 (b), 미결정·경계는 (c) fallback 견고함 Phase 1엔 과한 복잡도 중~높   추천 — (b) STAGE 4 자동 매핑 (Phase 1), (d)로 확장 여지 (Phase 2)
STAGE 4는 이미 유형 + 하객 + 예산 + 태그 + 양가를 종합해 추천 식장 형태(5성급 호텔 / 일반 웨딩홀 / 하우스 웨딩 / 레스토랑·스몰 베뉴 등)를 출력하고 있다. 이 출력을 그대로 베뉴 타입 enum으로 변환하면 추가 입력·질문·데이터가 전혀 필요 없다. 클라이언트 전용·이번 주 배포 제약에 가장 잘 맞는다.
type VenueType = 'hotel' | 'hall' | 'house' | 'small';

// STAGE 4 추천 식장 형태 문자열 → 베뉴 타입
function mapRecommendationToVenueType(recommendedForm: string): VenueType {
  const f = recommendedForm;
  if (/호텔|5성|특급/.test(f))                 return 'hotel';
  if (/일반\s*웨딩홀|컨벤션|예식장|대형/.test(f)) return 'hall';
  if (/하우스/.test(f))                         return 'house';
  if (/레스토랑|스몰|가든|채플|소규모/.test(f)) return 'small';
  return 'hall'; // 탐색미결정·미매칭 fallback (가장 보편적인 일반 웨딩홀)
}
한계 명시: STAGE 4는 "원하는/추천 형태"이지 "계약 형태"가 아니다. 질문설계 원칙(희망 ≠ 실제 선택)과 같은 갭이 존재한다. Phase 2에서 양가 압력 Modifier가 높을 때 한 단계 보수적으로(하우스→일반 등) 보정하는 (d) 혼합 규칙을 검토한다.

[추정] 표기 셀은 검증 자료가 없어 업계 통념 기반으로 채운 값. Phase 2에서 실측 교체 전까지 진단 밴드 비교에 사용하지 않는다(아래 D 참조).
계산 공식
interface VenueProfile {
  mealPrice: number;     // 1인 식대 (만원)
  guarantee: number;     // 보증인원 (명), 0이면 보증 없음
  rentalFee: number;     // 대관료 (만원), 호텔은 0
  noGuarantee: boolean;  // 스몰·하우스 협의형 여부
}

function calcVenueSubtotal(v: VenueProfile, guests: number) {
  // 식대: 보증 있는 베뉴는 max(하객, 보증) 기준, 보증 없으면 실하객 기준
  const billable = v.noGuarantee ? guests : Math.max(guests, v.guarantee);
  const meal = billable * v.mealPrice;

  // 대관료: 베뉴별 고정값 (호텔만 0)
  const rental = v.rentalFee;

  return { meal, rental, subtotal: meal + rental };
}
v2와의 차이는 한 줄이다. v2는 rental = guests >= guarantee ? 0 : (guarantee - guests) * mealPrice 였다(미달분을 대관료 명목으로만 부과, 초과 시 0). v3는 rental을 베뉴별 고정 대관료로 바꾸고 미달분은 식대(max(guests, guarantee))로 흡수한다. 이것이 실제 계약 구조(보증인원만큼 식대 지불 + 대관료 별도)와 일치한다.

C. 데이터 수집 plan
현황
현재 가격 DB(21지역 × 12개월 × 4카테고리 CSV)에는 베뉴 타입 구분과 대관료 컬럼이 없다. 예식장 카테고리가 단일 식대 단가만 갖고 있다.
우선순위
Phase 1 (즉시, 추정값) — 위 B 표의 단가·대관료를 코드 내 정적 상수 테이블(VENUE_PROFILES)로 하드코딩한다. 일반 웨딩홀·호텔은 검증 자료 기반이라 신뢰 가능, 하우스·스몰은 [추정]. 빌드 타임 정적이므로 CSV 스키마 변경 없이 바로 반영된다.
Phase 2 (배포 후, 실측 교체) — 베뉴 타입 × 지역 단가를 실측해 CSV로 이관.
출처 후보 (Phase 2 수집)
웨딩홀 견적 비교 사이트(아이웨딩·스마트웨딩 등) — 식대·대관료·보증인원 공개 견적
공여사들 / 다이렉트결혼준비 등 2026 가이드 — 항목별 평균
커뮤니티 실계약 데이터(Blind·디시·카페) — 실제 청구 금액(편향 주의, 다수 표본 필요)
매경·헤럴드 등 기사 — 식대 인상·호텔 프리미엄 동향
지역 차이
대관료·식대는 지역 편차가 크다(서울 강남 호텔 vs 광역시 호텔). Phase 1은 전국 단일값으로 시작하되, Phase 2에서 베뉴 타입 × M5 지역(서울/수도권/광역시/지방) 격자로 확장한다. 기존 지역계수(스드메에 쓰는 것)를 예식장에도 적용할지는 실측 후 결정 — 식대는 지역계수가 맞지만 대관료는 절대값 편차가 더 클 수 있어 별도 테이블 권장.

권장 — Phase 1엔 밴드 비교에서 대관료 분리
밴드 재조정은 실데이터가 모이는 Phase 2로 미루되(요청 선호와 일치), Phase 1에서는 표시용 금액과 진단용 core를 분리해 밴드 왜곡을 피한다.
// 표시용: 사용자가 보는 예식장 소계 — 대관료 포함 (정확한 금액)
const displaySubtotal = meal + rental;

// 진단용: 밴드 비교 core — 대관료 제외 (기존 밴드 그대로 사용, 분류 안정)
const diagnosisCore = coreWithoutRental; // v2와 동일 기준
이렇게 하면 Phase 1에서 사용자가 보는 금액은 정확해지고(대관료 누락 해결), 정합성 진단은 기존 밴드로 안정적으로 돌아간다. trade-off는 Phase 1 동안 진단이 대관료를 반영하지 못한다는 점인데, 분류 자체는 흔들리지 않으므로 수용 가능하다. Phase 2에서 대관료 포함 분포를 모아 밴드를 재산정하고 진단용 core에도 대관료를 넣는다.
Phase 2 밴드 옵션
옵션 1: 유형별 밴드 일괄 상향(표준실용 +200 등) — 단순
옵션 2: 베뉴 타입별 별도 밴드 — 정확하나 4유형 × 4베뉴 = 16밴드 관리 부담
권장: 옵션 1로 시작, 데이터가 충분하면 베뉴 타입을 밴드 보정 계수로 도입

E. 단계별 구현 plan
Phase 1 — 이번 주 배포 전 (최소 변경)
목표: 대관료 누락 갭만 막는다. 분류·밴드는 건드리지 않는다.
const VENUE_PROFILES: Record<VenueType, VenueProfile> = {
  hotel: { mealPrice: 10, guarantee: 200, rentalFee: 0,   noGuarantee: false },
  hall:  { mealPrice: 6,  guarantee: 200, rentalFee: 200, noGuarantee: false },
  house: { mealPrice: 8,  guarantee: 80,  rentalFee: 300, noGuarantee: true  },
  small: { mealPrice: 8,  guarantee: 30,  rentalFee: 100, noGuarantee: true  },
};

function patchVenueRental(recommendedForm: string, guests: number) {
  const vt = mapRecommendationToVenueType(recommendedForm);
  const v  = VENUE_PROFILES[vt];
  const { meal, rental, subtotal } = calcVenueSubtotal(v, guests);
  return {
    venueType: vt,
    displaySubtotal: subtotal,   // 대관료 포함 — 사용자 표시
    diagnosisCore: meal,         // 대관료 제외 — 기존 밴드 비교
    breakdown: { meal, rental },
  };
}

// 초소형 패치 버전 (매핑 함수도 생략)
if (/일반\s*웨딩홀|컨벤션|대형/.test(recommendedForm)) {

persona 분류 안정성
베뉴 타입은 STAGE 4(출력)에서 파생되고, persona 분류는 STAGE 1~2(축 A·B 점수)에서 결정된다. v3는 STAGE 5(예산 계산)만 수정하므로 분류 결과는 100% 불변이다. 같은 응답 → 같은 유형이 유지된다.
core 예산 변동 폭
평균: 일반 웨딩홀 케이스 +200만원 (전체 유저의 ~45%가 표준실용 → 가장 큰 모수)
