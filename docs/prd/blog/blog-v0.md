# Blog — 버짓로드 블로그

## Step 0. Feature Hierarchy
blog

## Step 1. User Goal
결혼 준비 중 생기는 궁금증과 예산 숫자의 근거를, 신뢰할 수 있는 가이드와 데이터를 통해 서비스 안에서 빠르게 해소하는 것

## Step 2. How This Helps
시뮬레이션이 "얼마"를 알려준다면, 블로그는 "그 숫자를 어떻게 이해하고 결정하면 되는지"를 알려준다.
결혼 준비 중 생기는 궁금증(용어 차이, 선택 기준, 항목별 의미)과 숫자의 근거(왜 이 금액이 나오는지)를 서비스 자체 가이드·데이터로 풀어줌으로써, 사용자가 외부 정보 탐색(오픈채팅, 포털, 남의 후기) 없이도 서비스 안에서 판단 근거를 얻게 한다.

## Step 3. Emotional Journey
"이 숫자가 무슨 의미지? 뭐부터 어떻게 고민해야 하지?" (막막함)
→ 추천 카드/목록에서 관련 글 발견: "어, 궁금하던 게 있네" (호기심)
→ 글 읽기 시작: "아, 이런 차이/의미였구나" (이해)
→ 데이터·기준 확인: "이래서 이런 금액이 나오는 거구나" (납득)
→ 글 읽기 완료: "이제 뭘 고민하면 되는지 알겠다" (확신 + 행동 동기)

## Step 4. Internal Feature Name
blog

## Step 5-0. Content Strategy

### 원칙
블로그는 **자체 생산 컨텐츠**로만 구성한다. 외부 블로그·카페·유튜브 등 타인 저작물의 인용·소개·재가공은 하지 않는다. 원본 링크 걸기 자체는 합법이나, 원저작자의 정서적 부담과 장기적 운영 리스크를 고려한 의도적 결정이다.

**허용되는 외부 소스**
- 공식 통계·데이터 (한국소비자원, 통계청 등) — 출처 링크 OK
- 언론 기사 — 출처 링크 OK
- 공공 도메인 자료

**금지되는 접근**
- 개인 블로그/카페/유튜브 후기의 소개·인용·재구성
- 업체/상품 개별 비교 리뷰 (정보 수집 비대칭 + 분쟁 리스크)

### 3가지 컨텐츠 축

**축 1 — 가이드·FAQ (알려주는)**
결혼 준비 중 검색하는 "○○가 뭐야? / 어떻게 해?"에 답하는 글. AI + 도메인 지식 + 편집으로 생산.
- 예: "스드메 스탠다드 vs 프리미엄, 뭐가 다를까"
- 예: "결혼 예산 짤 때 놓치는 숨은 비용 7가지"
- 예: "예물 vs 예단, 헷갈리는 두 용어 한 번에 정리"

**축 2 — 데이터 해설 (분석하는)**
한국소비자원·통계청·버짓로드 자체 데이터 기반 해설. 공식 출처 링크 가능.
- 예: "지역별 결혼 비용 — 강남이 지방보다 2배인 이유"
- 예: "컨벤션 대관료 1,360만 원 시대 — 2년간 얼마나 올랐나"
- 예: "버짓로드 유저 N명의 선택 TOP 5" (유저 데이터 쌓인 후)

**축 3 — 프레임·의사결정 (사고법 주는)**
"어떻게 결정하면 되는지"를 주는 메타 컨텐츠. AI 강점 활용.
- 예: "결혼 예산 우선순위 정하는 3가지 기준"
- 예: "양가 비용 분담, 싸우지 않고 정하는 체크리스트"
- 예: "견적서 받았을 때 꼭 확인해야 할 5가지"

### 단계별 전략

**초반 (런칭 ~ 유저 확보 전)** — **축 1 중심**
- 축 1(가이드·FAQ)로 3~5편 먼저 올림 — SEO 유입 + 검색 노출
- 원본 의존 0, 운영 공수 최소
- 목표: "결혼 예산 관련 궁금증은 버짓로드에서 해결된다"는 인상

**중반 (유저 쌓이기 시작)** — **축 2 추가 + 서비스 데이터 연결**
- 축 2(데이터 해설) 추가 — 팀이 조사한 한국소비자원 데이터, 버짓로드 자체 통계
- 서비스 소개와 데이터를 엮음 (예: "버짓로드는 어떤 데이터를 썼나 — 한국소비자원 + 공여사들 크로스체크")
- 차별성 강화 (경쟁사 복제 불가능한 자체 시각)

**장기 (유저 기반 자산 축적)** — **축 3 확대 + 유저 기반 사례**
- 축 3(프레임) 확대
- 동의받은 유저의 실사용 데이터 기반 사례 (자체 DB에서 추출)

### 초기 10편 믹스

| 비중 | 축 | 선택 이유 |
|---|---|---|
| 5편 | 축 1 (가이드·FAQ) | 만들기 쉽고 SEO 유입 잘 됨 |
| 3편 | 축 2 (데이터 해설) | 차별성 + 권위 + 팀 조사 데이터 재활용 |
| 2편 | 축 3 (프레임) | 공유/북마크 잘 됨 (바이럴 가능) |

## Step 5. Functional Requirements
1. 운영자는 Markdown으로 블로그 글을 작성하고 DB에 메타데이터(태그, 공개 여부)를 저장할 수 있어야 한다
2. 글에는 컨텐츠 축 태그를 부여할 수 있어야 한다 (필수, 1개):
   - `guide`: 가이드·FAQ — 결혼 준비 궁금증에 답하는 글
   - `data`: 데이터 해설 — 통계·자체 데이터 기반 분석
   - `frame`: 프레임·의사결정 — 사고법·결정 기준을 주는 글
3. 글에는 항목 태그(식장/스드메/예물/예단/신혼여행 등)를 부여할 수 있어야 한다
4. 사용자는 /blog 목록 페이지에서 전체 글을 탐색할 수 있어야 한다
5. 사용자는 /blog/{slug} 상세 페이지에서 글 전체를 읽을 수 있어야 한다
6. 목록은 태그별로 필터링할 수 있어야 한다
7. 해당 조건에 글이 없으면 관련 UI를 노출하지 않아야 한다
8. 결과 페이지 → 블로그 추천 카드는 사용자의 선택 항목(지역·예식장 유형·스드메 등급 등) 기반으로 관련 항목 태그를 매칭하여 정렬 우선순위를 적용한다 (경험 #5에서 상세 정의)

## Step 6. Existing vs New Screen
완전히 신규 화면 2개:
- /blog — 글 목록 페이지
- /blog/{slug} — 글 상세 페이지

## Step 7-2. Access Flow
진입점:
1. 헤더/GNB 메뉴 → /blog (목록)
2. 결과 페이지 하단 추천 카드 → /blog/{slug} (상세) ← 경험 #5에서 구현
3. 외부 검색 (SEO) → /blog/{slug} (상세)

흐름:
/blog (목록) → 글 클릭 → /blog/{slug} (상세) → 읽기 완료
→ 뒤로가기 → /blog (목록)
→ 또는 → /budget-draft (예산 시뮬레이션)

이탈점:
- 목록에서 원하는 글이 없을 때
- 상세 읽기 중 이탈
- 읽기 완료 후 다음 행동 없이 종료

## Step 8. Screen Inventory

### 화면 1: /blog — 목록 페이지
정보 위계:
1. 페이지 제목 + 소개 문구
2. 필터 (항목 태그 / 컨텐츠 축 태그)
3. 글 카드 목록 (썸네일 + 제목 + 요약 + 태그 + 작성일)
4. 글이 없을 경우 빈 상태 안내

### 화면 2: /blog/{slug} — 상세 페이지
정보 위계:
1. 뒤로가기 + 브레드크럼
2. 글 제목
3. 메타 정보 (작성일, 태그)
4. 본문 (Markdown 렌더링)
5. 하단 CTA (시뮬레이션 해보기 / 목록으로)
