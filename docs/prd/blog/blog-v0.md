# Blog — 사례 가이드 블로그

## Step 0. Feature Hierarchy
blog

## Step 1. User Goal
결혼 예산의 숫자가 실제로 어떤 결혼 모습인지를, 신뢰할 수 있는 사례를 통해 빠르게 감을 잡는 것

## Step 2. How This Helps
시뮬레이션이 "얼마"를 알려준다면, 블로그는 "그 돈으로 실제 어떤 결혼이 가능한지"를 보여준다.
모호한 숫자 범위(예: 3,000~5,000만 원)를 실제 사례로 치환해줌으로써, 사용자가 외부 정보 탐색(오픈채팅, 포털) 없이도 "내 예산이면 이 정도 결혼이 가능하구나"라는 확신을 서비스 안에서 얻게 한다.

## Step 3. Emotional Journey
"이 금액이면 뭘 할 수 있지?" (막막함)
→ 추천 카드 발견: "어, 비슷한 사례가 있네" (호기심)
→ 글 읽기 시작: "이 사람도 이 정도 예산이었구나" (공감)
→ 구체적 내역 확인: "아, 이렇게 구성하면 되겠다" (확신)
→ 글 읽기 완료: "나도 이 정도면 가능하겠다" (안도 + 행동 동기)

## Step 4. Internal Feature Name
blog

## Step 5. Functional Requirements
1. 운영자는 Markdown으로 블로그 글을 작성하고 DB에 메타데이터(태그, 공개 여부)를 저장할 수 있어야 한다
2. 글에는 예산 태그를 부여할 수 있어야 한다:
   - 전체 예산 태그: 총 예산 금액 (예: 3,500만 원) — 기본
   - 항목별 예산 태그: 전체 예산이 없는 경우, 특정 항목 금액 (예: 식장 1,200만 원)
3. 글에는 항목 태그(식장/스드메/예물/예단/신혼여행 등)를 부여할 수 있어야 한다
4. 사용자는 /blog 목록 페이지에서 전체 글을 탐색할 수 있어야 한다
5. 사용자는 /blog/{slug} 상세 페이지에서 글 전체를 읽을 수 있어야 한다
6. 목록은 태그별로 필터링할 수 있어야 한다
7. 해당 조건에 글이 없으면 관련 UI를 노출하지 않아야 한다
8. 글 개수가 늘어나면 예산 근접 매칭으로 정렬 우선순위를 적용할 수 있어야 한다

## Step 6. Existing vs New Screen
완전히 신규 화면 2개:
- /blog — 글 목록 페이지
- /blog/{slug} — 글 상세 페이지

## Step 7-2. Access Flow
진입점:
1. 헤더/GNB 메뉴 → /blog (목록)
2. 결과 페이지 하단 추천 카드 → /blog/{slug} (상세) ← 경험 #5에서 구현
3. 외부 검색 (SEO) → /blog/{slug} (상세)

흐름:
/blog (목록) → 글 클릭 → /blog/{slug} (상세) → 읽기 완료
→ 뒤로가기 → /blog (목록)
→ 또는 → /budget-draft (예산 시뮬레이션)

이탈점:
- 목록에서 원하는 글이 없을 때
- 상세 읽기 중 이탈
- 읽기 완료 후 다음 행동 없이 종료

## Step 8. Screen Inventory

### 화면 1: /blog — 목록 페이지
정보 위계:
1. 페이지 제목 + 소개 문구
2. 필터 (항목 태그 / 예산 구간)
3. 글 카드 목록 (썸네일 + 제목 + 요약 + 태그 + 작성일)
4. 글이 없을 경우 빈 상태 안내

### 화면 2: /blog/{slug} — 상세 페이지
정보 위계:
1. 뒤로가기 + 브레드크럼
2. 글 제목
3. 메타 정보 (작성일, 태그)
4. 본문 (Markdown 렌더링)
5. 하단 CTA (시뮬레이션 해보기 / 목록으로)
